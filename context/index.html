Delphie: A Code Context Engine for Agentic Workflows

Abstract

We present the results of SynSci Context Bench, an open-source benchmark harness for evaluating code context engines—the retrieval systems that feed relevant source code to AI coding agents. We compare Delphie (Synthetic Sciences), Context7, and Nia across 8 test suites comprising approximately 2,000 queries. The evaluation spans four phases of increasing rigor: hand-crafted custom benchmarks, industry-standard datasets (CoSQA, CodeSearchNet), blind LLM-as-Judge scoring, and a position-debiased enhanced judge with 4-dimensional scoring and RAGAS-inspired context quality metrics. Delphie achieves 2.8x higher scores than Context7 on code-to-code retrieval (CodeSearchNet) under position-debiased evaluation, winning 436 of 497 queries (88%). Context7 leads on documentation-style queries (CoSQA) by 1.67x, though we argue these queries have low ecological validity for agent workflows. All benchmark code, datasets, and raw results are publicly available.

01

Introduction

AI coding agents have become an integral part of modern software development. When an agent encounters an unfamiliar codebase, it needs to retrieve relevant source code, type definitions, function signatures, and implementation details to generate correct, context-aware responses. The system responsible for this retrieval is a code context engine.

Despite their importance, code context engines have lacked rigorous, standardized evaluation. Existing benchmarks either test general information retrieval (not code-specific), use proprietary datasets, or rely on simple keyword matching that fails to capture retrieval quality as perceived by an LLM consumer.

We built SynSci Context Bench to address this gap. The harness evaluates three publicly available context engines:

  • Delphie (Synthetic Sciences) — a scoped code retrieval engine that indexes specific repositories with AST-level extraction and context enrichment.
  • Context7 — a documentation retrieval system that serves pre-crawled library documentation and tutorials.
  • Nia — a universal knowledge search engine spanning code, documentation, papers, and packages across the ecosystem.

The evaluation progressed through four phases, each designed to address fairness concerns identified in the prior phase. This iterative approach is itself a contribution: we document the biases inherent in each evaluation methodology and how subsequent phases correct for them.

Our key findings are:

  1. Delphie dominates code-to-code retrieval. On CodeSearchNet (the industry standard), Delphie scores 2.8x higher than Context7 under position-debiased 4D evaluation, winning 88% of 497 queries.
  2. Context7 wins on documentation queries. On CoSQA, Context7 scores 1.67x higher—but these queries represent knowledge an LLM already has, making retrieval unnecessary in practice.
  3. Position debiasing is essential for code retrieval evaluation. Code chunks exhibit substantially stronger positional effects (Position Consistency = 0.553) than general text (0.82 baseline), confirming that single-pass LLM judging inflates scores for code retrieval by ~10%.
  4. Context enrichment provides measurable improvement. Post-retrieval enrichment with function signatures and docstrings improves LLM judge scores, with the largest gain in specificity (+0.082).

The entire benchmark harness, all datasets, and raw results are open-source at github.com/synthetic-sciences/SynSci-Context-Bench.

02

Delphie Architecture

Delphie is built around a simple principle: AI agents need code, not documentation. When an agent is navigating an unfamiliar codebase, it needs the actual function implementations, type definitions, and call sites—not a tutorial explaining how the library works. Delphie is purpose-built for this retrieval task.

2.1 Indexing Pipeline

Delphie indexes repositories at the chunk level rather than the file level. The indexing pipeline operates in four stages:

  1. Clone & parse. The repository is cloned and each source file is parsed into an AST (Abstract Syntax Tree). The AST identifies function definitions, class declarations, imports, type annotations, and call sites.
  2. Chunk extraction. Source files are split into semantically meaningful chunks using AST boundaries. Each chunk corresponds to a function, class, or logical code block rather than an arbitrary fixed-size window.
  3. Structural enrichment. Each chunk is enriched with structural metadata before embedding: file path, enclosing scope, defined symbols, imported modules, and relationships to adjacent chunks. This metadata is prepended to the chunk text before embedding computation.
  4. Embedding & storage. Enriched chunks are embedded using a code-optimized embedding model and stored in a vector database (Supabase) alongside the extracted symbol table and chunk relationship graph.
/* Delphie Indexing Pipeline */ Stage 1 Clone & Parse └─── Git clone → AST extraction → symbol table Stage 2 Chunk Extraction └─── AST-boundary splitting (functions, classes, blocks) └─── Preserves semantic coherence Stage 3 Structural Enrichment └─── File path, scope, symbols, imports └─── Adjacent chunk relationships (same_class, sequential) Stage 4 Embed & Store └─── Code-optimized embeddings (Gemini) └─── Vector DB (Supabase pgvector) + symbol index
Figure 1. Delphie's four-stage indexing pipeline. AST-boundary chunking and structural enrichment distinguish it from naive fixed-window approaches.

2.2 Search & Retrieval

At query time, Delphie performs a scoped vector similarity search against the indexed repository. Search is always scoped to a specific repository—the engine never returns results from unrelated codebases, eliminating the noise that plagues universal search approaches.

After retrieving the top-k chunks, Delphie performs post-retrieval enrichment: each chunk is augmented with the enclosing function or class signature, the first three lines of its docstring, and the last five lines of the preceding chunk for continuity. This enrichment runs two batched SQL queries (~10–30ms) and helps downstream LLM consumers understand each chunk without reading the entire file.

2.3 Architectural Differences

The three engines tested in this benchmark solve fundamentally different problems:

Aspect Delphie Context7 Nia
Design goal Scoped code retrieval within a repo Documentation retrieval Universal knowledge search
Search scope Specified repository only Pre-crawled docs/libs All indexed sources
Indexing Per-repo with AST extraction + enrichment Pre-crawled, no user indexing Global with doc crawling
Best use case “Find this function in my codebase” “How does library X work?” “Find knowledge across the ecosystem”
Content type Raw source code + signatures Docs, tutorials, guides Docs, code, papers, packages
Table 1. Architectural differences between the three evaluated engines. Each engine targets a different retrieval use case.

Benchmark Methodology

The evaluation was conducted in four phases, each designed to address fairness concerns from the prior phase. This layered approach ensures that no single evaluation methodology’s biases dominate the conclusions.

Phase Task Method Queries Engines
1 Custom retrieval quality Hand-crafted queries against FastAPI / Pydantic / httpx 55 Delphie, Nia, Context7
2 Industry-standard IR CoSQA + CodeSearchNet, content/file matching 997 Delphie, Nia
3 LLM-as-Judge (3D) Blind Claude Sonnet 4.6, 3 dimensions (rel/comp/spec) 997 Delphie, Context7, Nia
4 Enhanced Judge (4D, debiased) Position-debiased, 4 dimensions + faithfulness, RAGAS 997 Delphie, Context7
Table 2. Four-phase evaluation methodology. Each phase addresses fairness limitations of the prior one.

3.1 Why Four Phases?

Each phase was designed to correct for biases identified in the previous round:

  • Phase 1 used hand-crafted queries → potential bias toward Delphie’s design.
  • Phase 2 used industry-standard datasets → but content-matching penalizes engines that transform text during indexing.
  • Phase 3 used blind LLM judging → but single-pass scoring has ~10% positional bias (Zheng et al. 2023).
  • Phase 4 added position debiasing + faithfulness + judge consistency metrics → the most rigorous evaluation.

3.2 Statistical Rigor

Every pairwise comparison in the benchmark harness includes paired t-tests, Wilcoxon signed-rank tests, bootstrap confidence intervals (10,000 resamples), Cohen’s d (effect size), Cliff’s delta (non-parametric effect size), and Bonferroni correction for multiple comparisons. The enhanced judge additionally tracks its own reliability with Cohen’s kappa and Position Consistency metrics.

3.3 Test Corpora

Custom benchmarks use three well-known Python libraries as test corpora: FastAPI (web framework), Pydantic (data validation), and httpx (HTTP client). These were indexed on both Delphie and Nia; Context7 uses its pre-crawled documentation for these libraries. Industry-standard benchmarks use CoSQA (Huang et al. 2021) and CodeSearchNet (Husain et al. 2019) from HuggingFace.

3.4 Judge Configuration

All LLM-as-Judge evaluations use Claude Sonnet 4.6 (Anthropic) as the blind judge. The judge receives only the query and retrieved context—no engine identification, no metadata about which engine produced which result. In Phase 4, each query is evaluated twice with reversed chunk ordering; final scores are the average of both passes.

10 code generation tasks where engine-provided context is fed to an LLM, and the generated code is validated against known API surfaces. Delphie and Context7 both achieved a 40% hallucination rate; Nia scored 55.6%. Most hallucinations stemmed from outdated API references and invented method names in the model’s training data rather than retrieval failures.

05

Phase 2: Validated Retrieval

To address potential bias in hand-crafted queries, Phase 2 uses two industry-standard datasets from HuggingFace: CoSQA (Huang et al. 2021) and CodeSearchNet (Husain et al. 2019). Retrieval is measured by content similarity matching (Jaccard + SequenceMatcher, threshold 0.5) and file-path matching.

Why Context7 is excluded from Phase 2: Content-matching requires the engine to return text that closely matches a known ground-truth snippet. Context7 returns documentation rather than the actual code files in the benchmark corpus, so it cannot score hits. This is a limitation of the metric, not the engine. The LLM judge in Phases 3–4 bypasses this by evaluating usefulness regardless of source.

5.1 CodeSearchNet (497 queries)

The de facto standard for code search evaluation. Queries are natural language docstrings; ground truth is the corresponding function implementation.

Recall@10 0.949 0.000 0.053
NDCG@10 0.941 0.000 0.053
Table 7. CodeSearchNet validated retrieval. Delphie returns the correct function as the top result 93.8% of the time.

5.2 CoSQA (450 queries)

500 real web search queries for Python code with human-annotated relevance labels.

06

Phase 3: LLM-as-Judge

Phase 3 eliminates the content-matching bias of Phase 2 entirely. Instead of comparing retrieved text against a ground-truth snippet, a blind LLM judge evaluates whether the retrieved context is useful for answering the query—regardless of whether it matches a specific code snippet verbatim.

6.1 Methodology

  1. Send each query to each engine and collect raw retrieved context.
  2. Feed (query, context) pairs to Claude Sonnet 4.6 as a blind judge.
  3. Judge scores three dimensions (0–3 each): Relevance (does the context relate to the query?), Completeness (could you fully answer from this context?), and Specificity (is the context targeted, not generic?).
  4. Compare scores across engines; track wins and ties.

Completeness (0–3) 2.820 0.320 0.190

Total (0–3) 2.660 0.413 0.230
Wins 92 2 2
Table 9. LLM Judge on CodeSearchNet (3D, no debiasing). Delphie scores 6.4x higher than Context7. This is a single-pass evaluation—Phase 4 adds debiasing.

However, single-pass LLM judging carries known positional bias—Phase 4 corrects for this.

07

Phase 4: Enhanced Judge (Position-Debiased)

Phase 4 is the most rigorous benchmark in the suite. It improves on Phase 3 with three additions: position debiasing, a fourth scoring dimension (faithfulness), and judge self-consistency tracking.

7.1 Position Debiasing

LLM judges exhibit positional bias—they tend to rate chunks appearing first more favorably (Zheng et al. 2023). To correct for this, each query is evaluated twice: once with the original chunk ordering and once with the chunks reversed. The final score is the average of both passes. This eliminates the ~10% positional inflation documented in LLM evaluation literature.

7.2 Four-Dimensional Scoring

Phase 4 adds Faithfulness as a fourth scoring dimension, measuring whether the retrieved context is accurate and free of misleading information. Each dimension is scored 0–3:

Dimension Score 0 Score 1 Score 2 Score 3
Relevance No snippet relates Tangentially related Partially relevant Directly addresses query
Completeness Cannot answer Partial with gaps Mostly complete Fully answerable
Specificity Generic boilerplate Relevant + noise Mostly targeted Precisely targeted
Faithfulness Misleading/wrong info Some inaccuracies Mostly accurate Fully accurate
Table 11. Four-dimensional scoring rubric used by the enhanced judge. Each dimension is scored independently on a 0–3 scale.

Context Quality Metrics

In addition to LLM-based evaluation, we compute RAGAS-inspired context quality metrics that require no LLM calls—pure token-level analysis of retrieved context. These metrics provide an orthogonal view of retrieval quality.

Metric Definition
Context Precision Position-weighted relevance: chunks containing query terms score higher when ranked earlier
Context Density Fraction of retrieved tokens that contain query-relevant terms
Signal-to-Noise Ratio of useful content (code, identifiers) vs noise (whitespace, boilerplate)
Chunk Diversity 1 − avg pairwise Jaccard similarity (higher = more diverse results)
Table 15. RAGAS-inspired context quality metrics. Computed without LLM calls using pure token-level analysis.

8.1 Results

Metric CSN Delphie CSN Context7 CoSQA Delphie CoSQA Context7
Context Precision 0.481 0.191 0.491 0.235
Context Density 0.101 0.058 0.030 0.034
Signal-to-Noise 0.595 0.389 0.643 0.517
Chunk Diversity 0.945 0.925 0.927 0.938
Table 16. Context quality metrics across both datasets. Delphie has 2.5x better context precision on CodeSearchNet and 2.1x on CoSQA.

A notable finding: Delphie has 2.1x better context precision even on CoSQA, where it loses on judge scores. The issue is not retrieval precision—Delphie retrieves the right code—but content type mismatch. Code chunks do not answer documentation-style questions as well as tutorial content does, regardless of how precisely they are retrieved.

8.2 Judge Self-Consistency

Position debiasing allows us to measure how reliable the LLM judge itself is, by comparing its verdicts across both chunk orderings:

Metric CodeSearchNet CoSQA
Position Consistency 0.553 0.785
Cohen’s κ 0.290 (fair) 0.537 (moderate)
Avg Score Drift 1.089 0.382
N Evaluated 994 1,000
Table 17. Judge self-consistency metrics. CodeSearchNet has lower consistency (κ=0.290) because code chunks are highly order-sensitive.

The baseline Position Consistency for Claude-3.5-Sonnet on general text is 0.82±0.14 (Shi et al. 2025). Our code-retrieval PC of 0.553 on CodeSearchNet is substantially lower, confirming that code chunks exhibit stronger positional effects than natural language. This finding independently validates the need for position debiasing in code retrieval evaluation.

09

Context Enrichment

Delphie implements two layers of context enrichment that improve the usefulness of retrieved chunks for downstream LLM consumers.

9.1 Pre-Embedding Enrichment (Index Time)

Structural metadata is prepended to each chunk before embedding computation. This improves embedding quality by giving the embedding model more context about what the chunk represents:

# Structural metadata prepended before embedding # File: src/auth/middleware.py # Scope: AuthMiddleware > validate_token # Defines: validate_token, decode_jwt # Uses: import jwt, import datetime # After: refresh_token, revoke_token # Before: AuthMiddleware.__init__ <actual code chunk>
Figure 2. Pre-embedding structural enrichment. File path, scope, defined symbols, imports, and adjacent function names are included in the embedding input.

9.2 Post-Retrieval Enrichment (Search Time)

After retrieving the top-k chunks, each result is augmented with additional context from the database:

  • Enclosing function/class signature from the symbols table
  • Docstring (first 3 lines, truncated at 200 characters)
  • Preceding context (last 5 lines of the previous chunk for continuity)

This runs on at most top-k results (~10) using two batched SQL queries, adding only 10–30ms of latency.

9.3 Measured Impact

Dataset Without Enrichment With Enrichment Delta
CodeSearchNet Total 2.640 2.685 +0.045
CoSQA Total 1.167 1.181 +0.014
CodeSearchNet Specificity 2.300 2.382 +0.082
Table 18. Impact of post-retrieval enrichment on LLM judge scores. Specificity sees the largest improvement—function signatures help the judge confirm that the retrieved chunk is the right function.
10

Ecological Validity

A critical question overlooked by most retrieval benchmarks: do these queries reflect how AI agents actually use context engines?

10.1 CoSQA Queries Are Not Agent Queries

CoSQA queries look like web search queries: generic programming questions that any LLM can answer from training data alone. An agent would never invoke a context engine for these queries—it already knows the answer.

Real agent query (needs retrieval) CoSQA query (doesn’t need retrieval)
“Find where rate limiting is enforced in this repo” “python check file is readonly”
“What’s the schema for chunk_embeddings table?” “how to parse json from string in python”
“How does the SSE streaming handler work?” “python convert datetime to unix timestamp”
Table 19. Real agent queries vs CoSQA queries. The left column requires retrieval; the right column does not.

10.2 What This Means

  • CodeSearchNet (docstring → function) is closer to real agent use: it mirrors how agents navigate codebases to find specific implementations.
  • CoSQA scores should be weighted lower when evaluating context engines for agentic workflows.
  • Context7’s CoSQA advantage (1.67x) is real but applies to a use case where retrieval is largely unnecessary.
  • Delphie’s CodeSearchNet advantage (2.8x) applies to the use case where retrieval is essential—finding code in unfamiliar repositories.

10.3 Toward Higher-Validity Evaluation

We recommend future benchmarks use: (1) unfamiliar repositories the LLM has not seen in training, (2) task-completion metrics measuring whether the LLM can fix a bug or add a feature with retrieved context, and (3) SWE-Bench-style evaluation where context engine contribution is isolated while holding the LLM constant.

Limitations

These results should be interpreted with the following caveats:

  1. Custom datasets are not independent of the engine. Hand-crafted queries in Phase 1 were written by the team that built Delphie. There is inherent bias toward queries the engine handles well. This is why Phases 2–4 use independent datasets.
  2. Small sample sizes in custom benchmarks. 10–15 queries per benchmark can spot large differences but are not statistically robust. A 90% P@1 on 10 queries has a ±19% confidence interval (95% CI).
  3. Three repos do not represent all codebases. FastAPI, Pydantic, and httpx are well-structured Python libraries. Performance may degrade on messy codebases with poor documentation, mixed languages, or generated code.
  4. Nia’s API may not be optimally configured. Nia’s repositories filter does not appear to scope results to a specific repo, which is the root cause of its low scores. Their team may achieve better results with different configuration.
  5. Context7 was excluded from Phase 2. The content-matching metric requires indexed corpus files; Context7 returns documentation instead. This is a metric limitation addressed by Phase 3.
  6. Latency measured locally for Delphie. Delphie benchmarks ran with ~1.1s geographic latency to Supabase. Production latency (co-located on Render) is significantly lower.
  7. LLM judge may have biases. Claude Sonnet 4.6 may prefer certain response formats. However, the judge sees raw context with no engine identification, and position debiasing mitigates ordering effects.
  8. CoSQA weakness is structural. Delphie indexes code, not documentation. CoSQA queries are better answered by documentation than code. This is a design choice, not a bug.
12

Conclusion

We have presented the most comprehensive public evaluation of code context engines to date, spanning 8 benchmark suites, ~2,000 queries, and four phases of increasing methodological rigor. The key takeaways are:

  1. Delphie is the strongest engine for scoped code search—winning CodeSearchNet by 2.8x (debiased, 4D), 436 wins vs 36, and all custom benchmarks.
  2. Context7 wins on documentation-style queries—but these queries have low ecological validity for agent workflows. An LLM already knows how to parse JSON in Python and would never invoke a context engine for it.
  3. The engines solve different problems. Delphie excels at finding code in a specific repository (the actual agent use case), Context7 at answering how-to questions (a use case where retrieval is often unnecessary), and Nia at broad ecosystem search.
  4. Position debiasing is essential for code retrieval evaluation. Code chunks exhibit significantly stronger positional effects (PC=0.553) than general text (0.82 baseline). Any LLM-as-judge evaluation of code retrieval should use position debiasing.
  5. Context quality metrics reveal nuance. Delphie has 2.5x better context precision even on CoSQA where it loses on judge scores—the issue is content type mismatch, not retrieval quality.
  6. Post-retrieval enrichment provides measurable gains—+0.045 on CodeSearchNet total score, with specificity seeing the largest improvement (+0.082).

All benchmark code, datasets, and raw results are open-source. We welcome the community to reproduce these results, add new engines, and contribute to more rigorous evaluation of code context systems.

Open source: The full benchmark harness is available at github.com/synthetic-sciences/SynSci-Context-Bench. Add a new engine by implementing the ContextEngineAdapter interface in benchmarks/adapters/base.py.
